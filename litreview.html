Literature Review:

Mitigating Corporate Information Exposure on the Web

In the modern day, it is crucial for a business to have a presence on the internet. It has proven a valuable tool for finding new clients, and even as a base of operations for the company itself (such as e-commerce sites). With all the benefits, however, comes the large risk of opening the infrastructure of the company to the internet which is, by nature, insecure. In this review, I am going to research technical solutions to information exposure. Social engineering can play a big role in attacks, however, I am not going to cover the topic so that the technical aspects can be covered in more depth. Injection attacks are the number one web application security risk according to the Open Web Application Security Project (van der Stock, Smithline, & Gigler, 2017, p. 5). Between 2013 and 2017, injection has stayed in the top position and cross-site scripting (XSS) has fallen from third to seventh suggesting that the work done to prevent it is having an impact. I will be focussing on SQL injection and XSS due to the number of web applications vulnerable to injection and the likelihood of information being exposed by XSS attacks.

XSS is a common vulnerability in web applications, ranking as the 2nd most likely security vulnerability to affect users (Nayeem Khan, Johari Abdullah, Adnan Shahid Kahn, 2017, p. 169). The attacks can allow users to unknowingly execute malicious scripts or, more relevantly, allows session data to be stolen and can even expose encrypted SSL connections (Selvamani, Duraisamy, & Kannan, 2010, p. 230).

There are three categories of XSS attacks: persistent, where malicious code is stored in a database through a legitimate application (e.g. a post on a forum), DOM-injection and reflected/non-persistent; both of which let the browser execute malicious code (from response HTML for example). Each type requires a different approach to detect or prevent. A primitive method of detecting a persistent XSS attack is analysing the input data and performing basic checks, such as whether the string exceeds the character limit, followed by searching for specific tags to determine if a malicious payload is contained within (Das, Sharma, & Bhattacharyya, 2015, p. 813; Nayeem Khan, Johari Abdullah, Adnan Shahid Kahn, 2017, p. 171).

A client-side solution called Noxes (Kirda, Kruegel, Vigna, & Jovanovic, 2006) uses a firewall system to block unexpected connections, thus preventing stolen data from being sent back to the attacker. It works as a proxy that fetches requests on behalf of the browser and analyses the response for external links. This method only works to prevent data being stolen as the malicious code is still executed, so a denial of service attack (such as the browser window being closed) is not stopped.

Noncespaces is a more novel approach that uses randomization to completely remove XSS threats (Van Gundy & Chen, 2012). This technique randomises the HTML tags and their attributes in each document before the client displays the page. This allows the browser to detect injected code, as the attacker would not be able to know the random mapping of tags in that document, and therefore remove it before it executes.

JSand is a more general approach to XSS prevention by increasing the security of the JavaScript sandbox (Agten et al., 2012). It is a server-driven JavaScript sandboxing framework that requires no browser modifications and is enforced client side, making it easily deployable (unlike Noxes, for example). It isolates objects and functions, keeping track of all operations performed and checking them against a server-set security policy before allowing or blocking the operation.

Structured Query Language (SQL) injection attacks are another common vulnerability in web applications, specifically targeted at the databases connected to the app. Out of the three major injection attacks (SQL, OS, and LDAP), SQL remains the most common. There are several different forms of SQL injection attacks (SQLIAs), however, most of these are quite simple to prevent. These "classical"? forms of injection are all similar in how they hijack a legitimate SQL query to execute commands on the server or to gain information from the database. As such, they can mostly be avoided by sanitising input. A basic way of achieving this is to use prepared statements, SQL queries that have been precompiled with bind variables allowing for data to be input at a later time (Thomas & Williams, 2007, p. 2).

Another example of this is SAFELI, a query safety screen that checks for suspicious keywords in the query (like "or"? or "drop"?) along with escaping control characters such as the double and single quote (Fu & Qian, 2008, p. 34). Modern SQLIAs are more advanced and take advantage of vulnerabilities in the systems surrounding the database.

Amnesia is a tool that detects and prevents SQLIAs by monitoring the inputs of a system and building a model of all possible SQL queries that could be generated at each. At runtime, it compares the queries against the model and rejects any that violate it (Halfond & Orso, 2006, pp. 796–797). This model is not perfect and can generate false positives or negatives, however, it is fully automated.

Another automated form of protection against SQLIAs is RT-WASP (Salih & Samad, 2016). RT-WASP is an improvement of R-WASP and WASP (standing for Real-Time Web Application SQLI Protector). The tool uses "positive tainting"? to mark, in real time, the query strings as trusted at a character level. Before the query is sent to the database, evaluation of the string takes place and blocks all queries that contain one or more unmarked characters (that are part of keywords and operators).

A growing trend in modern technology is artificial intelligence and the rise of machine learning. This has been applied to SQLI detection using the Naïve Bayes

machine learning algorithm (Joshi & Geetha, 2014, p. 1112). The algorithm is trained on a large dataset of malicious and non-malicious queries, allowing it to calculate the likelihood of a new query being malicious and taking action if needed. It has proven to be relatively accurate (peaking at 93.3% accuracy) but is overshadowed by simpler alternatives such as Amnesia.

Unfortunately, there is no ultimate solution to the problem of information exposure. The proposed solutions I covered here all have their advantages, but with those come caveats and vulnerabilities of their own. Lasting protection requires active maintenance and a varied set of tools that can adapt to the ever-evolving attacks on the web.
